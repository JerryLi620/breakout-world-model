Debugging this world model surfaced a repeatable diagnostic ladder. The exact fixes are in the code/git history; what's durable is **how to read the symptoms** (these recur across architecture changes).

- **Static-after-few-frames / faint ball** in *teacher-forced* output is NOT rollout drift — teacher-forced re-feeds real frames each step, so a bad teacher-forced GIF means the *one-step* model is underfit. Fix one-step before touching rollout.
- **Mode collapse** (model emits a near-constant "mean" frame, `pred_max` identical every step, action sensitivity exactly 0.0) was the worst failure. Root cause was action being a tiny fraction of GRU input variance + a too-large `residual_scale` letting bricks saturate. Solved by **FiLM action conditioning** (action modulates every decoder block; impossible to optimize away) + smaller `residual_scale`.
- **Hedging** (pred_motion many× true_motion, faint ball smeared over candidate positions) is the deterministic-L1 signature. Tuned via `false_motion_weight` up / `change_weight` down. This is fragile knob-balancing — the principled fix is a non-L1 loss (gradient/edge, SSIM) or stochastic latent.
- One-step Breakout is **nearly deterministic** given true 4-frame stack + action, so L1 *can* fit it well; the L1-blur wall only truly bites in multi-modal *rollout*.

**Tools:** `diagnose.py` prints action-sensitivity (vary action on fixed input) and a per-step teacher-forced trace (pred_motion vs true_motion vs pred_max), and with `--save_gif` writes `diagnose_gt.gif` + `diagnose_teacher_pred.gif`. Always eyeball the GIF — healthy-looking numbers can still hide a blob.

**How to apply:** When predictions look wrong, first classify which of {underfit one-step, mode collapse, hedging, rollout drift} it is using diagnose.py, then act. See [[project-world-model]] for the overall goal.

The ladder to your goal:

Milestone Status Difficulty Notes
1. Crisp one-step (teacher-forced) ✅ done medium you are here
2. Action moves the paddle in rollout not started medium needs rollout training; should work
3. Short stable rollout (10–20 frames, ball stays alive) not started hard error compounding; ball intensity decay
4. Correct bounce physics (ball reflects off paddle/walls at right angle) not started hard must be learned from data, not guaranteed
5. Brick collisions (ball hits brick → brick vanishes → ball reflects) not started very hard causal, sparse events — easy to blur away
6. Long playable-looking rollout (50–100+ frames) not started research-grade where Dreamer/IRIS/DIAMOND earn their keep

The brutal truth about milestones 3–6: your model is deterministic pixel-L1. That class of model can usually reach milestones 2–3 and partially 4, but milestones 5–6 (sustained, physically-correct, brick-breaking gameplay over 100 frames) are where deterministic L1 hits a wall. The reasons:

  1. Error compounding. Each rollout step feeds the model its own output. Small errors snowball. A 1-pixel ball is the worst case — a tiny intensity decay per step and the ball vanishes within ~10 frames. You saw this exact collapse in your early runs.
  2. Sparse causal events. Brick-breaking happens rarely and depends on precise ball-brick contact. L1 averages, so it tends to fade bricks gradually instead of removing them on contact. Getting crisp "hit → gone" is genuinely hard.
  3. No mechanism to stay sharp over long horizons. This is exactly why SOTA uses discrete tokens (IRIS — each frame is a clean categorical pick, no decay) or diffusion (DIAMOND — re-sharpens every frame). Your residual model has no such mechanism.

Realistic outcomes for you:

  • Achievable next (weeks): Step-2 rollout training → paddle responds to actions, ball moves and bounces for ~10–20 frames before degrading. This already looks "game-like" for a short clip and is a solid course-project result.
  • Stretch (hard, maybe partial): ~30–50 frame rollouts with occasional correct bounces and some brick removal. Requires rollout training + more data + likely the gradient/edge loss for sharpness.

def save_rollout_gif(model, ds, args, device):
    """
    Free autoregressive rollout: start from the first num_frames REAL frames,
    then feed the model its own predictions back as input. Uses the recurrent
    hidden state (h carried forward) and the real action sequence.

    This is the honest test of whether the model can simulate forward. The
    per-step trace shows pred_motion (model's own step-to-step change) next to
    true_motion — when pred_motion decays toward 0 while true_motion stays
    positive, that's the frame where rollout collapsed.
    """
    from utils import tensor_to_numpy_frames, frames_to_gif

    seq = ds[args.gif_seq]["frames"].to(device)     # (T, 1, 64, 64)
    actions = ds[args.gif_seq]["actions"].to(device)
    T_len = seq.shape[0]

    frames_to_gif(tensor_to_numpy_frames(seq), "diagnose_gt.gif")

    rollout_frames = [seq[i] for i in range(args.num_frames)]   # real context
    current_stack = seq[: args.num_frames].unsqueeze(0).squeeze(2)  # (1, nf, 64, 64)
    h = model.init_hidden(1, device, seq.dtype)

    print("\n=== Free rollout trace ===")
    print("  i  action   pred_motion   true_motion   pred_max   (pred_motion->0 = collapse)")

    with torch.no_grad():
        for i in range(args.num_frames - 1, T_len - 1):
            prev = current_stack[:, -1:].clone()
            a = F.one_hot(actions[i : i + 1].long(), num_classes=args.action_dim).float()

            x_hat, _, _, h, _ = model(
                current_stack, action=a, h=h, deterministic=True, return_aux=True,
            )

            pred_motion = (x_hat - prev).abs().sum().item()
            true_motion = (seq[i + 1] - seq[i]).abs().sum().item()
            print(
                f"  {i:>2}  a={actions[i].item():<2}   "
                f"{pred_motion:>10.3f}   {true_motion:>10.3f}   "
                f"{x_hat.max().item():.3f}"
            )

            rollout_frames.append(x_hat.squeeze(0))
            current_stack = torch.cat([current_stack[:, 1:], x_hat], dim=1)

    roll_tensor = torch.stack(rollout_frames, dim=0)
    frames_to_gif(tensor_to_numpy_frames(roll_tensor), "diagnose_rollout.gif")

    print("\nSaved:")
    print("  diagnose_gt.gif")
    print("  diagnose_rollout.gif")
