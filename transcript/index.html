what's FiLM

FiLM = Feature-wise Linear Modulation. It's a simple, powerful way to let one input (here: the action) control how a neural network processes another input (here: the image features).

The core idea — one line of math:

Given a feature map h (your decoder's activations, shape [B, C, H, W]) and a conditioning vector e (your action embedding), FiLM produces a per-channel scale γ and bias β from e, then applies:

FiLM(h) = γ(e) · h + β(e)

γ and β are vectors of length C (one value per channel), broadcast across the spatial dimensions. So the action gets to say, channel by channel: "amplify this feature, suppress that one, shift this one up." It modulates the features rather than being concatenated alongside them.

Why it beats concatenation (your actual bug):

Your first attempt concatenated the 4-dim action onto the 256-dim latent. The action was <2% of the input, and the model learned to ignore it — action sensitivity was exactly 0.000. With FiLM, the action multiplies and shifts every feature map at every decoder stage. There's no way for the network to route around it — if the action changes, the features change. That's why sensitivity jumped to 0.20 after the switch.

In your code (models.py), the FiLM class:

gamma, beta = self.proj(e).chunk(2, dim=-1)   # action embed -> 2C numbers
return (1 + gamma) * feat + beta

Two details worth knowing: - (1 + gamma) instead of gamma: so γ=0 means "multiply by 1" = identity. Combined with the zero-initialized proj layer, the FiLM module starts as a no-op — at init the model behaves exactly as if FiLM weren't there, and can only learn to use the action. That's why it's safe to add: zero risk of breaking what already works. - It's applied at every decoder block (5 of them in your U-Net), so the action influences the output at every spatial resolution.

The intuition in one sentence: instead of telling the network the action (concatenation), FiLM lets the action reach in and turn the network's knobs — which is much harder to ignore.

It comes from Perez et al., 2018 ("FiLM: Visual Reasoning with a General Conditioning Layer"), originally for letting a text question condition image processing in visual question-answering — structurally the same problem as letting your action condition frame prediction.

How can we switch to stochastic/token model

Switching to a token model is the biggest single jump in your project, but it's well-defined. It's a two-stage architecture (this is IRIS in miniature). Here's the plan, what you reuse, and how to do it without drowning.

The architecture

Stage 1 — Tokenizer (VQ-VAE): turn each frame into discrete tokens.

frame 64×64×1  →  Encoder  →  8×8 grid of latent vectors
              →  Quantize (nearest codebook entry)  →  8×8 grid of token IDs
              →  Decoder  →  reconstructed 64×64 frame
  • A codebook of K entries (e.g. 512), each a learned vector.
  • Each of the 64 spatial cells picks its nearest codebook entry → 64 integer tokens per frame.
  • Trained on single frames: reconstruction loss + codebook/commitment loss, with the straight-through gradient trick.
  • Key property: tokens are discrete picks, so there's no intensity decay and no blur — the exact thing that killed your rollout.

Stage 2 — Dynamics (Transformer): predict next-frame tokens.

[frame_t tokens (64), action_t, frame_{t+1} tokens (64), action_{t+1}, ...]
  →  causal Transformer  →  predict next frame's 64 tokens
  • A small GPT-style causal transformer over the token sequence.
  • Trained with cross-entropy over the codebook (a classification problem, not regression — no L1 hedge).
  • Rollout = autoregressive sampling. Sample next tokens with a temperature → different samples give different futures. That's your stochasticity, for free, with the same action sequence.

will my training stop if my screen saver is triggered or my screen in off

No — neither a screensaver nor turning off the display stops your training. Those only affect what's drawn on screen; the CPU/GPU and your Python process keep running. The real risk on a MacBook is system sleep, which suspends all processes.

What actually stops it: - System sleep (lid closed, or idle long enough to sleep) → processes suspended, training pauses (and may not cleanly resume). - Closing the terminal/Terminal tab, or logging out → kills the process. - Battery dies.

What does NOT stop it: - Screensaver — safe. - Display turning off / "display sleep" — safe, training continues.

How to keep it running with the screen off:

The clean way is caffeinate. Either start training under it:

caffeinate -is python3 train.py ...your args...

-i prevents idle sleep, -s prevents sleep while on AC power. Or, if training is already running, caffeinate it by PID in another terminal:

caffeinate -is -w $(pgrep -f train.py | head -1)

That holds sleep off until the training process exits.

Important caveats for a MacBook: - Closing the lid still sleeps on most setups even with caffeinate (clamshell sleep), unless on AC power with an external display, or you've changed power settings. Safest: leave the lid open. - Keep it plugged in — caffeinate -s only blocks sleep on AC power. - Don't rely on "Prevent automatic sleeping" in Settings alone; caffeinate tied to the process is more reliable.

So: plug in, lid open, run under caffeinate -is, and you can let the display sleep freely. If you need to walk away and close the lid, that's when you'd want to move the job to Colab instead.

why the sample rollout so slow

Because token-by-token generation is the slow part of every autoregressive model, and my generate() is the naive version. Three compounding reasons:

1. The fundamental cost: it's autoregressive. A 16-frame rollout = 16 × 256 = ~4,096 sequential forward passes through the GPT. Each token needs the previous one, so they can't be parallelized. That's inherent to this model class.

2. No KV cache (the big fixable one). My generate() re-runs the entire transformer over the whole growing sequence for every single token. By late in the rollout the context is ~2,000 tokens × 8 layers, and you redo all of it just to produce one new token. That's O(L²) work repeated 4,096 times — almost all of it redundant recomputation of keys/values you already computed. A KV cache stores past keys/values so each step only processes the one new token. Typical speedup: 10–30×.

3. Python overhead + MPS. My loop uses Python lists and a for b in range(B) per token (clumsy), and MPS is slower than CUDA on these tiny per-step tensors. Minor next to #2, but real.

So a 3-sample × 16-frame run is ~12,000 forward passes with full recompute each — that's your couple-of-minutes.

Fixes, by payoff:

Fix Effort Speedup
Reduce --n_samples 1 while iterating trivial 3×
