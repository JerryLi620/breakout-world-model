Real progress — and a clear new failure mode.

Good news: - Action sensitivity is alive. |pred(a_i) - pred(a_j)| is 0.65–1.36 (sum) with max ~0.04–0.09 per pixel. FiLM is doing its job — different actions now produce different futures. - No more mode collapse. pred_max ≈ 0.58 matches GT max — the model isn't saturating bricks anymore. - No more static-after-frame-3. Predictions actually vary along the trace.

Bad news — new failure: hedging.

pred_motion ≈ 33–43 per step, but true_motion ≈ 0.4–16 (averaging ~5). The model is making 5–8× too much motion. Classic L1 hedge: instead of drawing one crisp ball at the predicted location, it draws faint balls across many candidate positions. The sum motion is high because the area of edits is large, even though each edit is small.

This is the mean-prediction blur I warned about, but expressed as area instead of intensity. With deterministic L1 and a single trajectory of training data, this is the model's safest bet.

Why the current loss allows it:

  • change_weight=500 rewards drawing something where the ball might be.
  • false_motion_weight=2 is way too weak to punish edits in regions where nothing changed.
  • The model finds: "draw the ball at 5 candidate positions, each at small intensity — change_weight rewards it for putting some mass in the right place, and false_motion only mildly penalizes the 4 wrong ones." Result: 5× motion.

Fix this with two knob changes — no architecture change needed:

  1. --false_motion_weight 50 (was 2). Make non-changed pixels actually expensive to touch.
  2. --change_weight 100 (was 500). Reduce the over-reward for drawing anything on change pixels — now the model has to draw at the right place, not anywhere plausible.
  3. (Optional) --residual_scale 0.2 (was 0.3). Cap each pixel edit smaller so the model can't fake a faint ball across a large region in one step.

Several levers, ordered by bang-for-buck. The first is free; the second is the real win.

1. Use the Apple GPU (MPS) — free, zero code change.

Your train.py already auto-detects it. Check the header line it prints:

Device : mps     ← good (Apple Silicon GPU)
Device : cpu     ← you're leaving 2–4× on the table

If it says cpu and you're on an M1/M2/M3, force it: --device mps. If it says cpu and you're on an Intel Mac, MPS isn't available and you skip this.

2. Vectorize the teacher-forced loop — the big one (~5–15×).

This is your real bottleneck. Right now the training loop does this per batch:

for t in range(n_steps):   # n_steps = 16 with seq_len=20
    x_hat = model(batch[:, t:t+4], ...)   # one forward + part of backward each

16 sequential forward passes, 16 Python-loop iterations, poor hardware utilization.

But — in your default config (--use_recurrent_teacher False, h=None every step), those 16 steps are completely independent. There's no recurrence linking them. So they can be collapsed into a single forward pass over a batch of B × n_steps = 32 × 16 = 512 examples. One forward, one backward, one big matmul that actually saturates the GPU/CPU vector units.

This is mathematically identical to what you're computing now (the loss is already a mean over steps), just far faster. It only works because rollout is off and the teacher is non-recurrent — exactly your Step-1 setup.

I can implement this. It changes only the teacher-forced section of the training loop; the rollout path (which must stay sequential because it's autoregressive) is untouched.

3. DataLoader workers — small win. num_workers=0 right now. Bump to 2–4 so data prep overlaps with compute. One-line change. (On macOS, num_workers>0 sometimes needs the if __name__ == "__main__" guard, which you already have.)

4. Shrink the iteration config — for debugging turnaround. While you're still tuning loss weights, you don't need the full run: - --num_sequences 500 (was 2000) → 4× fewer batches. - --epochs 8 (was 20). - --hidden_dim 256 (was 512) → smaller model.

This gives you a ~20-min sanity run instead of hours. Once weights are dialed in, scale back up for the final model.

5. Don't run the full thing on the MacBook at all. For the final 20–40 epoch runs, a free Colab T4 is ~10× your CPU. Develop/tune locally on the small config, then do the long final run on Colab.

My recommendation: do #1 (check MPS) and #4 (small config) right now for instant relief, and let me implement #2 (vectorized teacher-forced) — that's the structural fix that makes every future run fast, and it's safe because it's exactly equivalent to your current Step-1 math.
